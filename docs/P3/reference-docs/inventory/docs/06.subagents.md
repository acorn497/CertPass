# 서브 에이전트 구성

> 각 Phase의 구현을 담당할 서브 에이전트 배치 계획.

## 사용 에이전트 유형

| 에이전트 | 역할 | 담당 Phase |
|---------|------|-----------|
| `android` | Android 플랫폼 설정 (Gradle, AndroidManifest, 권한, 빌드 설정) | Phase 0 |
| `impl` | Flutter/Dart 기능 구현 (모델, DB, 상태관리, 화면, 위젯 코드 작성) | Phase 1 ~ 9 |
| `test` | 테스트 작성·실행 (단위 테스트, 위젯 테스트 작성 및 실행) | Phase 2 이후 각 Phase |

---

## 실행 계획

### Round 1 — 기반 설정 (병렬)

| 에이전트 | 작업 | 비고 |
|---------|------|------|
| `android` | Phase 0: AndroidManifest 카메라 권한, minSdk 확인, 앱 라벨 한글화 | 독립 작업 |
| `impl` | Phase 1: 의존성 설치, 디렉토리 구조, 테마, 라우팅, main.dart 리팩터링 | 독립 작업 |

> Phase 0과 Phase 1은 서로 의존성이 없으므로 **병렬 실행** 가능.

### Round 2 — 데이터베이스

| 에이전트 | 작업 | 비고 |
|---------|------|------|
| `impl` | Phase 2: drift 테이블, 모델(freezed), DB 클래스, Seed, DAO, 코드 생성 | Phase 1 완료 후 |

### Round 3 — DB 테스트 + 상태관리 (병렬)

| 에이전트 | 작업 | 비고 |
|---------|------|------|
| `test` | Phase 2 테스트: DB 스키마, Seed, DAO 단위 테스트 작성 및 실행 | Phase 2 완료 후 |
| `impl` | Phase 3: Riverpod providers, 비즈니스 로직 | Phase 2 완료 후 |

> DAO 테스트와 상태관리 구현은 **병렬 실행** 가능. 테스트는 DAO 레이어만 테스트하고, 상태관리는 DAO 위에 구축.

```
Round 1:  android(Phase 0) ─┬─ impl(Phase 1)
Round 2:                     └─ impl(Phase 2)
Round 3:  test(Phase 2)  ───┬─ impl(Phase 3)
Round 4:  test(Phase 3)  ───┬─ impl(Phase 4)
Round 5:  test(Phase 4)  ───┬─ impl(Phase 5)
Round 6:  test(Phase 5)  ───┬─ impl(Phase 6)
Round 7:  test(Phase 6)  ───┬─ impl(Phase 7)
Round 8:  test(Phase 7)  ───┬─ impl(Phase 8)
Round 9:  test(Phase 8)  ───┬─ impl(Phase 9)
Round 10: test(Phase 9)  ───┘
```

> Round 3부터는 **이전 Phase 테스트 + 다음 Phase 구현**을 병렬로 실행하여 처리량을 최대화한다.

## 에이전트별 공통 지침

### `android` 에이전트
- `android/` 하위 파일만 수정
- 빌드 설정 변경 후 `flutter analyze` 실행하여 검증

### `impl` 에이전트
- `lib/` 하위에 코드 작성
- `pubspec.yaml` 수정 시 `flutter pub get` 실행
- 코드 생성 필요 시 `dart run build_runner build --delete-conflicting-outputs` 실행
- 기존 l10n 문자열(`AppLocalizations`) 적극 활용
- UI 목업(`docs/ui/*.html`)과 ERD(`docs/02.erd.md`) 참조

### `test` 에이전트
- `test/` 하위에 테스트 작성
- 테스트 계획(`docs/05.test_plan.md`)의 테스트 케이스 기준
- DB 테스트는 `sqflite_common_ffi` 사용 (메모리 DB)
- 바코드 스캔 등 하드웨어 의존 기능은 mock 처리
- 테스트 작성 후 `flutter test` 실행하여 통과 확인
